Wildfire Risk Ranking from Satellite Data

A near-real-time system that pulls NASA satellite hotspot data, clusters detections into fire events, and scores each one for how likely it is to grow into a declared incident. Most tools tell you where fires are. This one tries to tell you which ones are about to get worse.

PythonNASA FIRMSVIIRSST-DBSCAN geopandasshapelyXGBoostFastAPI PostgreSQL/PostGISRedisDagsterMLflow PydanticDocker

Problem

On a bad fire weather day, the satellite feed shows 20, 30, 40 independent hotspot clusters at once. Most public tools (Watch Duty, WFIGS, FIRMS viewers) tell you where fires are. None of them tell you which ones are about to blow up before they're large enough to generate an ICS-209 incident report.

Satellite detections can lead official reporting by hours to days. The goal is to use that window: score clusters as they form and flag the ones most likely to outrun containment.

Approach

Raw VIIRS hotspot detections from NOAA-20 and S-NPP go through an eight-stage pipeline, each stage an isolated module with a typed data contract. Dagster orchestrates the DAG - it tracks which artifacts are current and re-materializes only the affected downstream stages when logic or data changes, which matters when re-running clustering on 1.24M rows.

Incoming rows are validated by a Pydantic model hierarchy (Raw → Clean → DB) and quarantined on failure rather than dropped - preserving the original payload for debugging. Clean rows are deduplicated by 1 km spatial grid and 15-minute time bins, filtered to vegetation fires only (type = 0), then partitioned into 1°×1° tiles before ST-DBSCAN clusters them into fire events. Daily cluster snapshots are gap-filled to account for satellite pass gaps. Features are computed from those snapshots and labeled against ICS-209-PLUS records using a three-way distance split, then assembled into the training matrix.

System

The live system is a modular monolith: a single deployable unit where each pipeline stage is fully isolated behind a typed data contract. Dagster orchestrates the asset graph - tracking what was materialized, when, and with what parameters - so any stage can be re-run in isolation without touching the others. Postgres/PostGIS stores every pipeline artifact, Redis caches the serving layer, and MLflow manages every training run and model promotion.

Design decisions

Dagster over Airflow/Prefect. Each pipeline stage produces a named, typed artifact, which maps naturally to Dagster's asset model. If clustering logic changes, only the stages downstream of clustering re-run; ETL and preprocessing stay cached.

Rolling-window clustering. ST-DBSCAN re-runs over a sliding window each cycle rather than incrementally updating existing clusters. Satellite passes shift hotspot positions slightly between observations, so incremental assignment would accumulate positional drift; a clean re-run avoids that.

Three-way label split. ICS-209 only files reports above a complexity threshold (~100–300 acres). A cluster that burned out below that threshold never generates a report, but it can't safely be called a true negative either. Ambiguous clusters are left unlabeled and excluded from training.

PR-AUC over accuracy. Class imbalance is severe (~1–5% positive rate). A model that always predicts "not significant" hits 95%+ accuracy and is useless. Precision-recall AUC is the right metric for a system where the cost of missing a fast-moving fire far exceeds the cost of a false alarm.
