Wildfire Risk Ranking from Satellite Data

A near-real-time system that pulls NASA satellite hotspot data, clusters detections into fire events, and scores each one for how likely it is to grow into a declared incident. Most tools tell you where fires are. This one tries to tell you which ones are about to get worse.

Problem

On a bad fire weather day, the satellite feed shows 20, 30, 40 independent hotspot clusters at once. Most public tools (Watch Duty, WFIGS, FIRMS viewers) tell you where fires are. None of them tell you which ones are about to blow up before they're large enough to generate an ICS-209 incident report.

Satellite detections can lead official reporting by hours to days. The goal is to use that window: score clusters as they form and flag the ones most likely to outrun containment.

Approach

Raw VIIRS hotspot detections from NOAA-20 and S-NPP go through an eight-stage pipeline, each stage an isolated module with a typed data contract. Dagster orchestrates the DAG - it tracks which artifacts are current and re-materializes only the affected downstream stages when logic or data changes, which matters when re-running clustering on 1.24M rows.

Incoming rows are validated by a Pydantic model hierarchy (Raw → Clean → DB) and quarantined on failure rather than dropped - preserving the original payload for debugging. Clean rows are deduplicated by 1 km spatial grid and 15-minute time bins, filtered to vegetation fires only (type = 0), then partitioned into 1°×1° tiles before ST-DBSCAN clusters them into fire events. Daily cluster snapshots are gap-filled to account for satellite pass gaps. Features are computed from those snapshots and labeled against ICS-209-PLUS records using a three-way distance split, then assembled into the training matrix.

Plan

Phase 1 - Data Exploration
EDA across all four data sources: VIIRS hotspot detections, ICS-209-PLUS incident records, WFIGS fire perimeters, and FIRMS archive.

Phase 2 - Data Engineering Pipeline
Eight-stage pipeline from raw FIRMS detections to labeled training matrix.

  • Stage 1a - ETL / Validation: ingest FIRMS rows, validate with Pydantic, quarantine failures
  • Stage 1b - Preprocessing: dedup by 1 km grid and 15-min bins, filter to vegetation fires (type = 0)
  • Stage 2 - Spatial Partitioning: tile into 1°×1° cells with 0.1° overlap
  • Stage 3 - ST-DBSCAN Clustering: cluster detections into fire events (ε₁ = 1 km, ε₂ = 48 h <- to start with)
  • Stage 4 - Aggregation: build daily cluster snapshots, gap-fill satellite pass gaps
  • Stage 5 - Feature Engineering: compute FRP, growth rate, spread, area, perimeter proximity
  • Stage 6 - Labeling: match clusters to ICS-209-PLUS records via three-way distance split
  • Stage 7 - Feature Matrix Assembly: assemble train (2018–2020) and test (2023) splits

Phase 3 - Pipeline Module Implementation
Takes validated logic from each stage's EDA notebook and consolidates it into clean, importable Python modules. Goal: a single command that runs the full pipeline on raw parquet inputs and produces X_train, y_train, X_test as output artifacts.

Phase 4 - Risk Model
XGBoost classifier trained on the assembled feature matrix, optimized for PR-AUC given severe class imbalance (~1–5% positive rate).

Phase 5 - Serving
FastAPI service with Redis cache layer exposing scored fire events in near-real-time.

System

The live system is a modular monolith: a single deployable unit where each pipeline stage is fully isolated behind a typed data contract. Dagster orchestrates the asset graph - tracking what was materialized, when, and with what parameters - so any stage can be re-run in isolation without touching the others. Postgres/PostGIS stores every pipeline artifact, Redis caches the serving layer, and MLflow manages every training run and model promotion.

Design decisions

Dagster over Airflow/Prefect. The pipeline produces a named, typed artifact at each stage, which maps directly to Dagster's asset model. Every stage is tracked: what was materialized, when, from what inputs, and with what parameters. If clustering logic changes, Dagster re-materializes from Stage 3 without re-running ETL and preprocessing upstream. Airflow's DAG-first model is less natural for artifact-centric pipelines; Prefect is a reasonable alternative but lacks the built-in asset catalog and lineage graph that replace a lot of custom bookkeeping.

Five-schema Postgres design. Each schema has a distinct role. staging is append-only: rows are never modified after insert, and validation failures are quarantined with their full Pydantic error payload rather than dropped. pipeline holds cleaned artifacts from each stage. labels holds ICS-209 ground truth, loaded once. api contains materialized views refreshed after each pipeline run, optimized for low-latency reads. monitoring tracks prediction outcomes for the feedback loop. Keeping these separated prevents the operational store from leaking into the analytical store and makes the access patterns for each schema explicit.

Pipeline-driven Redis invalidation. The pipeline runs every ~3 hours; without caching, every GET /fires/active request triggers a spatial join across hundreds of clusters. On pipeline completion, the active fires key is deleted and a version integer is incremented atomically. Nearby-query cache keys embed the version prefix, so a single increment invalidates all of them without a slow SCAN + DELETE loop. A 3-hour TTL acts as a safety net if the invalidation call fails: the cache self-heals within one FIRMS cycle.

Rolling-window clustering. ST-DBSCAN re-runs over a 7–14 day sliding window each cycle rather than incrementally appending to existing clusters. Satellite passes shift hotspot positions slightly between observations, so incremental assignment would accumulate positional drift over time. A clean re-run is simpler and avoids that state management complexity for v1.

Three-way label split. ICS-209 only files reports for fires above a complexity threshold (~100–300 acres depending on jurisdiction). A cluster that burned 80 acres and was quickly contained never generates a report — but it can't safely be called a true negative either. Forcing a negative label on those cases corrupts training. Clusters within 10–50 km of an ICS-209 origin are left unlabeled and excluded.

XGBoost and PR-AUC. The feature set is tabular and the class imbalance is severe (~1–5% positive rate). XGBoost handles imbalance natively via scale_pos_weight and tolerates nullable CV feature columns without imputation — useful since the CV pipeline phase isn't built yet. A model that predicts "not a significant fire" for everything achieves 95%+ accuracy and is useless; precision-recall AUC is the right metric for a system that needs high recall at a controlled false positive rate.

Status

Complete: EDA across all four data sources; ~1.24M CA VIIRS hotspots downloaded and spot-checked; ST-DBSCAN parameters tested against the DOE fire (450k acres, Aug 2020); labeling strategy, feature definitions, and train/test split finalized.

In progress: Data engineering pipeline (preprocessing through feature matrix assembly).

Planned: XGBoost risk model → FastAPI serving layer → Docker deployment. There's also an optional CV branch that pulls features from raw VIIRS thermal IR imagery (Band I4) via a lightweight CNN. It's designed as an additive step so it can be introduced later without changing downstream schemas.
